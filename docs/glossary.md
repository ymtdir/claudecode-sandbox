## 2. ビジネス用語

### タスク (Task)
**定義**: ユーザーが完了すべき作業単位。タイトル、期限、優先度などの属性を持つ。

**属性**:
- タイトル（必須）
- 説明（任意）
- 期限（任意）
- 優先度（低/中/高）
- カテゴリー
- ステータス（未完了/完了/アーカイブ）

**例**: 「会議資料の作成」「買い物リスト」

**関連用語**: Todo、リマインダー

---

### カレンダー (Calendar)
**定義**: 時間軸上でタスク、イベント、リマインダーを表示・管理するビュー。

**種類**:
- 月表示: 1ヶ月分のカレンダーグリッド
- 週表示: 7日間の時間軸表示
- 日表示: 1日の詳細タイムライン

**関連用語**: タイムライン、スケジュール

---

### リマインダー (Reminder)
**定義**: 特定の時刻または場所でユーザーに通知を送る機能。

**種類**:
- 時間ベース: 指定時刻に通知
- 位置ベース: 特定の場所で通知（ジオフェンシング）
- 繰り返し: 定期的に通知

**例**: 「毎朝8時に薬を飲む」「会社に着いたら資料印刷」

**関連用語**: 通知、アラート

---

### 統合ビュー (Unified View)
**定義**: カレンダー、タスク、リマインダーを一つの画面で表示する、本アプリの中核機能。

**特徴**:
- 全ての情報を時間軸上に統合表示
- アプリ間の切り替え不要
- 一元的な管理が可能

**関連用語**: 統合カレンダー、一元管理

---

### タスク完了率 (Task Completion Rate)
**定義**: 一定期間内で完了したタスクの割合。ユーザーの生産性指標。

**計算式**: `(完了タスク数 / 全タスク数) × 100`

**表示場所**: 統計ダッシュボード、デイリービュー

**関連用語**: 生産性、達成度

---

### ストリーク (Streak)
**定義**: タスクを連続して完了した日数。ゲーミフィケーション要素の一つ。

**例**: 「7日連続でタスク完了」

**目的**: ユーザーのモチベーション維持

**関連用語**: 連続記録、習慣トラッカー

---

### タイムブロッキング (Time Blocking)
**定義**: カレンダー上の時間枠にタスクを割り当てる時間管理手法。

**操作**:
- ドラッグ&ドロップで時間割り当て
- 時間枠の調整
- 自動配置提案（将来機能）

**関連用語**: 時間管理、スケジューリング

---

### 共有カレンダー (Shared Calendar)
**定義**: 複数のユーザーが閲覧・編集できるカレンダー。

**権限レベル**:
- 閲覧のみ: 見るだけ
- 編集可能: タスクの追加・編集が可能
- 管理者: メンバー管理も可能

**用途**: 家族やチームでの共同利用

**関連用語**: コラボレーション、共有設定

---

### プライベートタスク (Private Task)
**定義**: 共有カレンダーで他のユーザーに表示されないタスク。

**用途**: 個人的な予定の保護

**表示**: 自分のデバイスでのみ表示

**関連用語**: プライバシー設定

---

## 3. 技術用語

### React Native
**定義**: Facebook（現Meta）が開発したクロスプラットフォームモバイルアプリケーションフレームワーク。

**バージョン**: 0.73+

**特徴**:
- iOS/Android両対応
- JavaScriptでネイティブアプリ開発
- ホットリロード対応

**関連用語**: クロスプラットフォーム、ネイティブアプリ

---

### TypeScript
**定義**: Microsoftが開発したJavaScriptの静的型付け拡張言語。

**バージョン**: 5.9+

**利点**:
- 型安全性
- IDE支援の向上
- リファクタリングの容易さ

**関連用語**: 静的型付け、型定義

---

### Redux Toolkit
**定義**: Reduxの公式ツールセット。状態管理を簡潔に記述できる。

**主要概念**:
- Store: アプリケーション全体の状態
- Slice: 状態の一部分
- Action: 状態変更の指示
- Reducer: 状態変更のロジック

**関連用語**: 状態管理、グローバルステート

---

### Expo
**定義**: React Native開発を簡素化するツールチェーンとプラットフォーム。

**バージョン**: SDK 50+

**機能**:
- 開発環境の簡易セットアップ
- OTA（Over-the-Air）アップデート
- ビルドサービス（EAS）

**関連用語**: EAS Build、Expo Go

---

### Firebase / Supabase
**定義**: BaaS（Backend as a Service）プラットフォーム。

**提供サービス**:
- 認証（Authentication）
- データベース（Firestore/PostgreSQL）
- ストレージ（Cloud Storage）
- プッシュ通知（FCM）

**関連用語**: BaaS、クラウドサービス

---

### オフラインファースト (Offline-First)
**定義**: ネットワーク接続がない状態でも動作を優先する設計思想。

**実装方法**:
- ローカルDBへの即時保存
- バックグラウンド同期
- 楽観的更新

**関連用語**: 同期、ローカルストレージ

---

### API (Application Programming Interface)
**定義**: アプリケーション間でデータをやり取りするためのインターフェース。

**種類**:
- REST API: HTTPベースの通信
- GraphQL: クエリ言語ベースのAPI

**関連用語**: エンドポイント、HTTP、JSON

---

### CI/CD
**定義**: 継続的インテグレーション/継続的デプロイメント。

**ツール**: GitHub Actions

**プロセス**:
- CI: コード統合時の自動テスト
- CD: テスト成功後の自動デプロイ

**関連用語**: 自動化、ビルドパイプライン

---

### JWT (JSON Web Token)
**定義**: 認証情報を安全に伝送するためのトークン形式。

**構成**:
- Header: トークンのタイプと暗号化方式
- Payload: ユーザー情報
- Signature: 検証用の署名

**用途**: ユーザー認証、API認可

**関連用語**: 認証、トークン

---

## 4. プロセス用語

### スプリント (Sprint)
**定義**: 開発作業の固定期間（通常2週間）。

**構成要素**:
- スプリント計画
- デイリースタンドアップ
- スプリントレビュー
- レトロスペクティブ

**関連用語**: アジャイル、スクラム

---

### プルリクエスト (Pull Request / PR)
**定義**: コード変更をメインブランチに統合するためのレビュー依頼。

**プロセス**:
1. feature ブランチで開発
2. PR作成
3. コードレビュー
4. 承認とマージ

**関連用語**: コードレビュー、マージ

---

### MVP (Minimum Viable Product)
**定義**: 最小限の機能を持つ製品。早期リリースと検証が目的。

**UnifiedCalのMVP機能**:
- 基本カレンダー表示
- タスク管理
- 基本リマインダー

**関連用語**: プロダクト開発、リリース

---

### KPI (Key Performance Indicator)
**定義**: 重要業績評価指標。プロダクトの成功を測定する指標。

**UnifiedCalのKPI**:
- ダウンロード数
- 7日間継続率
- タスク完了率
- NPS（Net Promoter Score）

**関連用語**: 成功指標、メトリクス

---

### テストカバレッジ (Test Coverage)
**定義**: テストコードがカバーしているソースコードの割合。

**目標値**:
- ユニットテスト: 80%
- 統合テスト: 60%

**測定ツール**: Jest、Coverage Report

**関連用語**: 品質保証、自動テスト

---

### Git Flow
**定義**: Git を使用したブランチ管理戦略。

**ブランチ構成**:
- main: 本番環境
- develop: 開発統合
- feature/*: 機能開発
- release/*: リリース準備
- hotfix/*: 緊急修正

**関連用語**: バージョン管理、ブランチ戦略

---

### コードレビュー (Code Review)
**定義**: 他の開発者によるコード変更の確認プロセス。

**チェックポイント**:
- 機能性
- 可読性
- パフォーマンス
- セキュリティ

**関連用語**: 品質管理、ピアレビュー

---

### デプロイメント (Deployment)
**定義**: アプリケーションを実行環境に配置するプロセス。

**環境**:
- Development: 開発環境
- Staging: テスト環境
- Production: 本番環境

**関連用語**: リリース、環境

## 6. 用語の使用ガイドライン

### 6.1 一貫性の原則
- ドキュメント全体で同じ用語を使用する
- 同義語は避け、定義された用語を使用する
- 新しい用語を導入する場合は、用語集に追加する

### 6.2 ユーザー向け用語
- 専門用語を避け、分かりやすい表現を使用
- 例: 「同期」より「データの更新」
- UI上の表記は日本語を優先

### 6.3 技術文書での用語
- 正式名称を使用（略語は初出時に説明）
- コード内では英語表記を使用
- コメントは日本語も可

### 6.4 用語の更新
- 新しい用語や概念が追加された場合、速やかに用語集を更新
- 変更履歴を記録
- チーム全体に変更を通知
